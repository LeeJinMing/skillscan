# SkillScan Homepage IA

这份文档不是文案，而是首页信息架构。  
目标是回答两个问题：

1. 首页要先讲什么，后讲什么
2. 每一屏要让用户明白什么

---

## 中文版

## 页面目标

首页不是拿来讲全部细节的。  
首页只要完成 4 件事：

1. 让人知道你卖什么
2. 让人知道你不是 scanner 小工具
3. 让人知道你先从 `skills` 切入，但空间更大
4. 让目标用户愿意点“预约演示”或“查看文档”

## 页面结构

### 第一屏：Hero

**目标**  
10 秒内讲清定位。

**必须回答**

- 你是什么
- 你管什么
- 你不是啥

**建议内容**

- Headline
- Subheadline
- Supporting line
- 2 个 CTA
- 1 张流程图或产品示意图

### 第二屏：问题

**目标**  
把用户从“扩展扫描”拉到“扩展治理”。

**必须回答**

- 企业为什么会痛
- 他们为什么不能只靠 scanner

**建议形式**

- 一句总述
- 5-7 个治理问题 bullet

### 第三屏：产品定义

**目标**  
讲清你是两层结构。

**必须回答**

- Scanner 做什么
- Governance 做什么

**建议形式**

- 两栏卡片
- 或一条从扫描到审批的流程线

### 第四屏：核心能力

**目标**  
让用户一眼看出你卖的是控制权。

**建议放 5 张卡**

- 准入门禁
- 审批闭环
- 版本复审
- 隔离策略
- 审计追溯

### 第五屏：为什么先从 skills 开始

**目标**  
把“切口窄”和“空间大”同时讲清。

**必须回答**

- 为什么是 `skills`
- 为什么未来不止 `skills`

### 第六屏：工作流

**目标**  
把产品用法讲清，不展开技术细节。

**建议步骤**

1. 扫描
2. 报告
3. verdict
4. 审批
5. 发布 / 安装门禁

### 第七屏：适合谁

**目标**  
让访客自我代入。

**建议对象**

- 平台团队
- 安全团队
- 合规团队
- 想把 agent 带进企业场景的产品团队

### 第八屏：为什么不是 scanner

**目标**  
主动化解“你是不是只是一个扫描器增强版”的质疑。

**建议形式**

- 左边：Scanner
- 右边：SkillScan

对比维度：

- 发现风险
- 能否阻止进入生产
- 是否有审批闭环
- 是否支持版本复审
- 是否有审计链

### 第九屏：CTA

**目标**  
明确下一步动作。

**建议按钮**

- 预约演示
- 查看 GitHub
- 查看文档

## 页面顺序原则

正确顺序：

1. 定位
2. 问题
3. 产品定义
4. 能力
5. 为什么从 `skills` 开始
6. 工作流
7. 适合谁
8. 对比
9. CTA

不要这样排：

- 一上来就讲技术实现
- 一上来就讲支持很多对象
- 一上来先讲 scanner 规则细节

## English Version

## Page Goal

The homepage should not explain everything.  
It only needs to do four jobs:

1. Make visitors understand what you sell
2. Make them understand this is not just a scanner utility
3. Show that you start with `skills` but have broader platform scope
4. Drive them toward “Book a demo” or “Read the docs”

## Page Structure

### Section 1: Hero

**Goal**  
Explain positioning in 10 seconds.

**Must answer**

- What are you
- What do you control
- What are you not

**Suggested content**

- Headline
- Subheadline
- Supporting line
- Two CTAs
- One product flow visual

### Section 2: Problem

**Goal**  
Move the reader from “extension scanning” to “extension governance.”

**Must answer**

- Why does this hurt enterprises
- Why is a scanner alone not enough

### Section 3: Product Definition

**Goal**  
Show the two-layer model clearly.

**Must answer**

- What the scanner does
- What governance does

### Section 4: Core Capabilities

**Goal**  
Make it obvious that you are selling control, not just detection.

**Suggested five cards**

- Admission control
- Approval workflow
- Version-aware review
- Isolation policy
- Audit trail

### Section 5: Why start with skills

**Goal**  
Explain both the narrow wedge and the larger scope.

**Must answer**

- Why `skills`
- Why not only `skills`

### Section 6: Workflow

**Goal**  
Show how the product works without going too deep technically.

**Suggested steps**

1. Scan
2. Report
3. Verdict
4. Approval
5. Release / install gate

### Section 7: Who it is for

**Goal**  
Help the visitor self-identify quickly.

**Suggested audiences**

- Platform teams
- Security teams
- Compliance teams
- Product teams bringing agents into enterprise workflows

### Section 8: Why this is not just a scanner

**Goal**  
Answer the obvious objection early.

**Suggested layout**

- Left: scanner
- Right: SkillScan

Comparison dimensions:

- Detects risk
- Blocks entry into production
- Has approval workflow
- Supports version-aware review
- Provides auditability

### Section 9: CTA

**Goal**  
Make the next action obvious.

**Suggested buttons**

- Book a demo
- View on GitHub
- Read the docs

## Ordering principle

Correct order:

1. Positioning
2. Problem
3. Product definition
4. Capabilities
5. Why start with `skills`
6. Workflow
7. Audience
8. Comparison
9. CTA

Avoid this order:

- Starting with technical implementation
- Starting with broad object coverage
- Starting with scanner rule details
